# Gluten 鲲鹏构建方案架构文档

## 1. 概述

本文档描述 Gluten（向量化加速库）在鲲鹏 ARM 架构上的自动化构建方案，重点介绍缓存策略的设计与实现。

### 1.1 构建目标

- 在 ARM 架构上编译 Gluten 及其依赖
- 产出可部署的软件包 `gluten.zip`
- 通过缓存机制优化构建时间

### 1.2 技术栈

| 组件 | 版本/说明 |
|-----|----------|
| Runner | `ubuntu-22.04-arm` (原生ARM环境) |
| 构建容器 | `openeuler22.03_lts_sp3:arm64_003` |
| 构建工具 | Maven 3.9.9, CMake, GCC |
| CI平台 | GitHub Actions |

### 3.2 Maven 依赖缓存

#### 3.2.1 缓存配置

```yaml
- name: Cache Maven dependencies
  uses: actions/cache@v4
  with:
    path: maven-docker-cache          # Runner主机上的缓存目录
    key: ${{ runner.os }}-arm-maven-${{ github.run_id }}
    restore-keys: |
      ${{ runner.os }}-arm-maven-     # 匹配最近的缓存
```

#### 3.2.2 目录挂载机制

**Runner主机目录** → **容器内目录** 映射：

```yaml
docker run -d \
    -v "${{ github.workspace }}/maven-docker-cache:/root/.m2" \
    ...
```

| Runner 路径 | 容器路径 | 说明 |
|------------|---------|------|
| `maven-docker-cache/` | `/root/.m2/` | Maven本地仓库 |
| - | `/root/.m2/repository/` | 所有下载的jar包 |
| - | `/root/.m2/settings.xml` | Maven配置文件 |

#### 3.2.4 缓存Key策略

```
Key格式: Linux-arm-maven-{run_id}
Restore Keys:
  - Linux-arm-maven-{run_id}     (精确匹配，通常不存在)
  - Linux-arm-maven-             (前缀匹配，找最新的)
```

**示例**：

| 构建 | Key | 恢复的缓存 | 新增内容 |
|-----|-----|----------|---------|
| #100 | `Linux-arm-maven-100` | 无 | 全量下载 |
| #101 | `Linux-arm-maven-101` | `Linux-arm-maven-100` | 增量下载 |
| #102 | `Linux-arm-maven-102` | `Linux-arm-maven-101` | 增量下载 |

### 3.3 Docker 镜像缓存

#### 3.3.1 缓存配置

```yaml
- name: Cache Docker image
  uses: actions/cache@v4
  with:
    path: docker-image-cache
    key: ${{ runner.os }}-arm-docker-openeuler22.03-arm64_003
```

#### 3.3.2 加载/拉取逻辑

```bash
IMAGE="swr.cn-north-4.myhuaweicloud.com/cloud_boostkit/openeuler22.03_lts_sp3:arm64_003"
CACHE_FILE="${{ github.workspace }}/docker-image-cache/openeuler.tar"

if [[ -f "$CACHE_FILE" ]]; then
    # 缓存命中：从tar文件加载镜像
    docker load -i "$CACHE_FILE"
else
    # 缓存未命中：从SWR拉取镜像
    docker pull "$IMAGE"
    # 保存镜像到缓存目录
    mkdir -p docker-image-cache
    docker save -o "$CACHE_FILE" "$IMAGE"
fi
```

#### 3.3.3 工作流程

```
┌────────────────────────────────────────────────────────────────────┐
│                   Docker镜像缓存工作流程                            │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│              ┌─────────────────────┐                               │
│              │ openeuler.tar 存在? │                               │
│              └──────────┬──────────┘                               │
│                    ┌────┴────┐                                     │
│                    │         │                                     │
│                   Yes       No                                     │
│                    │         │                                     │
│                    ▼         ▼                                     │
│           ┌───────────┐  ┌───────────────┐                         │
│           │ docker    │  │ docker pull   │  (~3分钟)               │
│           │ load -i   │  │ 从SWR拉取镜像 │                         │
│           │ (~30秒)   │  └───────┬───────┘                         │
│           └───────────┘          │                                 │
│                                  ▼                                 │
│                          ┌───────────────┐                         │
│                          │ docker save   │                         │
│                          │ 保存为tar文件 │                         │
│                          │ (~5.3GB)      │                         │
│                          └───────────────┘                         │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

#### 3.3.4 缓存Key特点

- **固定Key**：`Linux-arm-docker-openeuler22.03-arm64_003`
- **命中条件**：相同镜像tag的构建复用缓存
- **失效时机**：镜像tag更新（如 `arm64_003` → `arm64_004`）

## 4. 不使用缓存的组件

### 4.1 Git 代码仓库

**原因**：

1. **避免权限问题**：缓存恢复后目录所有权变化，导致 `git` 报 `safe.directory` 错误
2. **确保代码最新**：每次克隆保证获取最新代码
3. **简化逻辑**：无需处理分支切换、冲突等复杂情况

**当前实现**：

```bash
# 直接克隆，不使用缓存
git clone -b $gluten_branch https://gitcode.com/openeuler/Gluten.git /opt/gluten
```

### 4.2 编译中间产物

**原因**：

1. 编译产物与代码强相关，缓存可能导致不一致
2. 完整编译更可靠，避免增量编译的潜在问题

## 7. 缓存管理

### 7.1 缓存生命周期

- GitHub Actions 缓存默认 **7天** 未访问后过期
- 缓存总限制：**10GB** per repository
- 超出限制后自动删除最旧的缓存

### 7.2 手动清理缓存

```bash
# 通过 GitHub CLI 清理缓存
gh cache delete --all

# 清理特定缓存
gh cache delete Linux-arm-maven-12345
```

### 7.3 缓存Key设计建议

| 场景 | Key策略 | 说明 |
|-----|--------|------|
| 频繁更新的依赖 | `{os}-arm-maven-{run_id}` | 每次保存，restore恢复最新 |
| 稳定的镜像 | `{os}-arm-docker-{tag}` | 固定key，镜像更新才失效 |
| 基于内容的缓存 | `{os}-{hash}` | 内容变化才更新 |

## 8. 故障排查

### 8.1 常见问题

| 问题 | 可能原因 | 解决方案 |
|-----|---------|---------|
| 缓存未命中 | 7天未访问已过期 | 触发一次构建 |
| Maven依赖下载失败 | 网络问题/仓库不可用 | 检查网络连接 |
| Docker镜像加载失败 | 缓存文件损坏 | 删除缓存重新构建 |
| 磁盘空间不足 | 缓存文件过大 | 清理旧缓存 |

### 8.2 调试命令

```bash
# 查看缓存大小
du -sh maven-docker-cache/
du -sh docker-image-cache/

# 查看Maven仓库内容
ls -la maven-docker-cache/repository/

# 验证Docker镜像

```
